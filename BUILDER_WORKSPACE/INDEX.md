## LAYER 4 — EXECUTION BINDING: Build Rules

### RULE 1 — ARCHITECTURE BEFORE CODE (L20)
Read and internalize the full architecture. Every file. Every model. Every equation.
Never write code that you do not understand the doctrine behind.

### RULE 2 — ANCIENT MATH ONLY (L18)
Every constant must trace back to one of: φ, Fibonacci, Schumann, π, e, Planck h, speed of light c.
No arbitrary numbers. No magic numbers. No configuration constants that are not φ-derived.
If a threshold appears without an ancient math derivation, it is a build error.

### RULE 3 — 4-LAYER ARTIFACTS ALWAYS
Every new model, engine, law, or document must be written as a 4-layer MEDINA-ARTIFACT:
```
LAYER 1 — MEANING:           Doctrine clause. What this IS. One sentence.
LAYER 2 — MODEL:             Typed schema. All fields. Units. Ranges. Simultaneous roles.
LAYER 3 — COMPUTATION:       All state equations. All formulas. All ancient forms.
LAYER 4 — EXECUTION BINDING: Engine name. Function. Gate. Beat condition.
```
See phi.mo lines 196–241 (PHI_SOVEREIGN) for the canonical artifact format.

### RULE 4 — RESIDENTS VS COMPUTATES
**Residents** (in phi.mo, types.mo, doctrine files): declare, never mutate
**Computates** (in heartbeat, cognition_layer, engines): execute every 873ms, read residents
Never put mutable `var` state inside a module — Motoko M0014. State belongs to the actor.
Pass state into module functions as parameters. Return updated state.

### RULE 5 — MOTOKO CONSTRAINTS
- `var` state: only in actor (`main.mo` stable vars). Never in modules.
- Nat64 arithmetic: `(val).toNat().toFloat()` — dot notation only
- Float functions: `Float.cos()`, `Float.sqrt()`, `(f).toInt()` — check which is dot notation
- No function overloading (M0051) — all function names must be unique in the actor
- Import mo:core only — no mo:base. Map from `mo:core/Map`, not `mo:base/OrderedMap`

### RULE 6 — ZERO-EXPOSURE WALL (L24 + L27)
Before any public query function is written, ask: "Does this expose a doctrine name, law label, or model name?"
If yes: return only the numeric output, the proof hash, or the coherence score.
Public interfaces return: `Float`, `Nat`, `Text` (as a hash or numeric string), `Bool`.
Never return: law names, model names, engine names, doctrine strings.
`ZERO_EXPOSURE` check is enforced in every heartbeat.

### RULE 7 — ADDING NEW LAWS
New laws must:
1. Be assigned the next available number (current: L49 is max)
2. Have an `absoluteAnchor` pointing to one of A01..A20
3. Have an `enforcementFn` that is a real function in laws.mo
4. Be added to the `LAWS` array in phi.mo
5. Be documented as a 4-layer artifact in BUILDER_WORKSPACE

### RULE 8 — ADDING NEW MODELS
New models must:
1. Be declared in types.mo with all 4 simultaneous roles commented (cognitive + economic + geometric + proof)
2. Have a builder function in types.mo (`buildXxx(...)`)
3. Have a backward-compatibility alias if replacing an existing type
4. Be added to N1-SOVEREIGN-MODEL-HIERARCHY.md under the correct N1 parent

### RULE 9 — ADDING NEW ENGINES
New engines must:
1. Live in a separate `.mo` file if they exceed 50 lines
2. Accept state as parameters — never hold `var` state
3. Return updated state, not void
4. Be registered in the cognition layer's 11-engine array
5. Fire every 873ms without exception

### RULE 10 — THE LOOP NEVER CLOSES (L23)
Every output PARALLAX produces must flow back into the artifact feedback pipeline.
`artifact_feedback.mo` is not optional. If a production event does not trigger re-ingestion, the organism is not eating its own light. That is a build error.
